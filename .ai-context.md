# SVGnest Project Context

## Project Overview

**SVGnest** is a TypeScript/Rust hybrid browser-based vector nesting tool for optimizing polygon packing. It's a modernized and
performance-enhanced fork of the original SVGnest, restructured as a monorepo with significant architectural improvements.

### Key Features & Improvements

-   **Monorepo Structure**: Uses Lerna for managing multiple packages
-   **TypeScript Migration**: Converted from JavaScript to TypeScript for better type safety
-   **WebAssembly Integration**: Critical performance sections migrated to Rust/WASM (10x speed improvement)
-   **Memory Optimization**: Migrated from Float64 to Float32/Int32 for better performance
-   **Worker Optimization**: Workers are created once and reused with ArrayBuffer I/O
-   **Modern React UI**: User interface built with React and modern tooling
-   **Isolated Dependencies**: Custom js-clipper implementation to reduce bundle size

## Package Details

### 1. wasm-nesting

**Purpose**: High-performance Rust/WASM module for critical calculations and nesting orchestration

-   **Language**: Rust with wasm-bindgen
-   **Key Components**:
    -   `WasmPacker`: Singleton managing nesting state (genetic algorithm, NFP store)
    -   `clipper`: Full Clipper implementation in Rust for offset and boolean operations
    -   `genetic_algorithm`: Population-based optimization for part placement
    -   `nesting`: NFP calculation, placement algorithms, and cache management
    -   `geometry`: Point, Polygon, BoundRect with Float32 precision
-   **Key WASM Functions**:
    -   `wasm_nest()`: Full synchronous nesting flow in Rust (no workers)
    -   `wasm_packer_init()`, `get_pairs()`, `get_placement_data()`, `get_placement_result()`: Step-by-step API
    -   `calculate_chunk_wasm()`: Worker-compatible chunk processing
    -   Bit manipulation and utility functions
-   **Data Format**: Uses `Vec<Vec<f32>>` internally; flat Float32Array with u32-bit-encoded size prefixes for JS interop

### 2. polygon-packer

**Purpose**: Main packing algorithm orchestration with parallel worker support

-   **Main Class**: `PolygonPacker`
-   **Features**:
    -   Orchestrates full nesting workflow (init → pairs → NFPs → placements → results)
    -   Parallel worker management via `Parallel` class
    -   Chunked data distribution to workers for NFP/placement calculations
    -   Configuration serialization (curveTolerance, spacing, rotations, populationSize, mutationRate, useHoles)
    -   Progress tracking and display callbacks
-   **Data Helpers**:
    -   `joinFloat32Arrays()`: Encodes arrays with u32 size prefixes for WASM
    -   `splitFloat32Arrays()`: Decodes flat arrays back to per-item chunks
    -   `mergeFloat32Arrays()`: Concatenates multiple ArrayBuffers
-   **Dependencies**: wasm-nesting (workspace), Parallel (worker orchestration)

### 3. svg-parser

### Data Exchange Format (JS ↔ WASM)

**Encoding Convention**: Polygons and chunks are passed as flat `Float32Array` where each vector is prefixed by its size encoded
as a u32 bit pattern inside an f32 slot.

**JavaScript Side**:

```typescript
// Encode: Write u32 size bits into Float32Array using DataView (little-endian)
const dv = new DataView(buffer);
dv.setUint32(byteOffset, size, true); // true = little-endian

// Decode: Read u32 size bits from Float32Array using DataView
const size = dv.getUint32(byteOffset, true);
```

**Rust Side**:

```rust
// Encode: Convert u32 size to f32 bit pattern
flat.push(f32::from_bits(size as u32));

// Decode: Read f32 and extract u32 bits
let size = size_f.to_bits() as usize;
```

**Helpers**:

-   `split_f32_chunks(&[f32]) -> Vec<Vec<f32>>`: Parse flat array with size prefixes
-   `join_f32_chunks(&Vec<Vec<f32>>) -> Vec<f32>`: Flatten vectors with size prefixes

## Key Algorithms

### Nesting Algorithm Flow

**Two Modes Available**:

1. **Worker-based (Parallel)**: TypeScript orchestrates parallel worker execution

    - `PolygonPacker.start()` → init WASM → get pairs → distribute chunks to workers → compute NFPs in parallel → get placement
      data → compute placements in workers → get placement result
    - Uses `Parallel` class to manage worker pool
    - Chunks pairs/placements for distribution (default chunk size: 512)

2. **Synchronous (Rust-only)**: Full flow executed in WASM without workers
    - `wasm_nest(config, polygons, chunk_size)` → runs entire nesting in Rust: init → pairs → NFPs → placement data → placements
      → result bytes
    - No JavaScript workers; all computation in single WASM call
    - Useful for Node.js or non-browser environments

**Core Steps**:

1. **Initialization**: Parse polygons, generate bounds/tree (Clipper offset), initialize genetic algorithm
2. **Pair Generation**: Genetic algorithm produces individual → extract NFP pairs to calculate
3. **NFP Calculation**: For each pair (bin/part or part/part), compute No Fit Polygon using Minkowski sum
4. **Placement Data**: Update NFP cache, serialize placement problem state
5. **Placement Calculation**: Use NFPs to find valid positions for parts on bins
6. **Result Serialization**: Best fitness, placed parts, bounds, source tree

### Genetic Algorithm

-   **Singleton**: `GeneticAlgorithm` (thread_local in Rust)
-   **Operations**: Crossover, mutation, fitness evaluation
-   **Fitness**: Minimizes bin area usage and maximizes parts placed
-   **Population**: Configurable size (default encoded in config)

### NFP Store

-   **Singleton**: `NFPStore` (thread_local in Rust)
-   **Caching**: HashMap of computed NFPs (key: hash of pair)
-   **Serialization**: `serialize_map_to_f32()` encodes cache as flat f32 array (no sorting for performance)

### Clipper Integration

-   **Full Rust Implementation**: `packages/wasm-nesting/src/clipper/`
-   **Operations**: Offset (spacing), boolean ops (union, difference, intersection), polygon simplification
-   **Data Locality**: Custom implementation optimized for cache performance

### Performance Optimizations

-   **WASM Integration**: 10x+ speed improvement for NFP pair calculations
-   **Memory Layout**: Data locality patterns for better cache performance (custom Clipper)
-   **Typed Arrays**: Float32Array for geometry operations (reduced from Float64)
-   **Worker Pools**: Persistent workers to avoid creation overhead; ArrayBuffer-based I/O
-   **Chunking**: Large workloads split into chunks for parallel distribution
-   **No Sorting**: Removed expensive HashMap key sorting in serialization
-   **Singleton Pattern**: thread_local RefCell for WasmPacker, GeneticAlgorithm, NFPStore (avoids recreation)

## Common Development Tasks

### Adding WASM Exports

1. Define function in `packages/wasm-nesting/src/lib.rs` with `#[wasm_bindgen]`
2. Use `split_f32_chunks` / `join_f32_chunks` for data conversion
3. Rebuild wasm: `cd packages/wasm-nesting && npm run build`
4. Add TypeScript wrapper in `polygon-packer/src/wasm-nesting.ts`
5. Rebuild TypeScript: `cd packages/polygon-packer && npm run build`

### Modifying Nesting Logic

1. Update Rust code in `packages/wasm-nesting/src/nesting/` or `genetic_algorithm/`
2. Test with `cargo check` and `cargo test` in `packages/wasm-nesting/`
3. Rebuild wasm bindings
4. Update `WasmPacker` or `PolygonPacker` if API changes

1. Coordinate changes between Rust helpers (`split_f32_chunks`, `join_f32_chunks`) and TypeScript helpers (`splitFloat32Arrays`,
   `joinFloat32Arrays`, `mergeFloat32Arrays`)
2. Ensure u32 size encoding/decoding matches (f32::from_bits / .to_bits in Rust; DataView setUint32/getUint32 in JS)
3. Update worker-flow if chunk structure changes

### UI Enhancements

2. Update i18n files for internationalization
3. Add shared styles as needed

### Performance Optimization

4. Optimize data serialization (minimize copies, use views where possible)

## Important Notes for AI Agents

### Data Encoding Rules

-   **Always** use u32 bit patterns in f32 slots for integer sizes when passing data between JS and WASM
-   **JavaScript**: Use `DataView.setUint32(offset, size, true)` and `DataView.getUint32(offset, true)` (little-endian)
-   **Rust**: Use `f32::from_bits(size as u32)` to encode and `size_f.to_bits() as usize` to decode

### Architecture Patterns

-   **Singletons**: `WasmPacker`, `GeneticAlgorithm`, `NFPStore` use `thread_local! { static INSTANCE: RefCell<...> }`
-   **Separation**: TypeScript orchestrates workflow; Rust handles computationally intensive operations
-   **Workers**: Managed by `Parallel` class; receive/send ArrayBuffer chunks

### Code Organization

-   **Rust modules**: Keep related functionality grouped (clipper/, genetic_algorithm/, nesting/, geometry/)
-   **WASM exports**: Minimal wrapper functions in `lib.rs`; delegate to internal modules
-   **TypeScript wrappers**: Thin layer in `wasm-nesting.ts` calling WASM; orchestration in `polygon-packer.ts`

This context should help agents understand the project structure, make appropriate modifications, and maintain consistency with
the existing architecture and patterns.
